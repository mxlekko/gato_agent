# 前端页面与后端接口映射表

## 1. 文档目标

本文档对应 `FE0-T2`，用于冻结：

- 每个前端页面依赖哪些后端接口
- 每个接口的最小请求 / 响应结构
- 哪些页面可以先用 mock 数据推进
- 哪些页面必须真实对接后端能力

本文档是给前端独立推进用的，不要求当前所有接口已经存在。

## 2. API 设计原则

前端控制台 V1 统一遵守以下原则：

1. 浏览器只能调用 API 服务层
2. 浏览器不能直连内部 Helper / ModelTool / Gateway / DirectDb 服务
3. 只保留一个现有真实联调入口：
   - `POST /api/agent/run`
4. 新增控制台只读 / 调试 / 配置接口统一走：
   - `/api/console/*`
5. 所有控制台接口复用统一 envelope：

```json
{
  "success": true,
  "requestId": "req_xxx",
  "data": {},
  "error": null
}
```

错误 envelope 复用当前 API 结构：

```json
{
  "success": false,
  "requestId": "req_xxx",
  "data": null,
  "error": {
    "code": "INVALID_REQUEST",
    "message": "xxx",
    "httpStatus": 400,
    "stage": "request-validate",
    "retryable": false,
    "details": null
  }
}
```

## 3. 前端禁止直连的内部接口

前端页面禁止直接调用以下内部地址：

- `http://127.0.0.1:19101/internal/context/sales-opportunity` ContextHelper
- `http://127.0.0.1:19102/internal/directdb/sales-opportunity` DirectDbRunner
- `http://127.0.0.1:19103/internal/model/validate-structured-output` ModelTool
- `http://127.0.0.1:19104/internal/rag/search` RAG

这些能力只能由后端 API 或控制台聚合接口代为调用。前端也不应配置或探测任何已退役的本地 agent gateway。

## 7. 前端开发的最小约束

前端开发可据此独立推进时，必须满足：

1. 页面只依赖本文档列出的 API facade
2. 浏览器不直接访问内部 tool 服务
3. 页面层统一处理 `success / requestId / data / error` envelope
4. 在真实接口未就绪时，mock 数据形状必须与本文档保持一致

## 8. FE0-T2 完成判定

完成 `FE0-T2` 时，应满足：

1. Scene 列表、Workflow 详情、Run 详情、Trace 详情、配置校验等页面都有明确接口
2. 每个页面都标明了最小数据结构
3. 每个页面都标明了 mock / real 边界
4. 前端可以据此开始独立开发，不必等后端一次性把所有接口实现完
