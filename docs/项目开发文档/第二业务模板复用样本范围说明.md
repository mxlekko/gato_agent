# 第二业务模板复用样本范围说明

## 1. 样本选择结论

阶段 `P6-T1` 选择 `sales-opportunity-advisor-directdb` 作为第二业务模板复用验证样本。

对应现有场景文件：

- `scene-configs/sales-opportunity-advisor-directdb.json`

对应最小请求样例：

- `tests/fixtures/reuse-candidate/sales-opportunity-advisor-directdb.request.json`

## 2. 选择理由

选择 `sales-opportunity-advisor-directdb` 的原因是它与当前已经完成并行迁移的 `sales-opportunity-advisor` 在输入、输出、业务目标和主体流程上高度相似，但在“数据获取方式”上有明确差异，正适合作为模板复用验证样本。

它满足下面几个条件：

1. 输入结构相近  
   两个场景都只要求 `bizParams.opportunityId`。

2. 输出结构相近  
   两个场景都要返回“销售机会推进建议”结构化结果，字段形态保持一致，核心仍是 `summary / adviceText / nextActions / basisFields`。

3. 业务目标相同  
   都是在“已有销售机会事实基础上生成推进建议”，只是底层上下文获取链路不同。

4. 差异足够集中  
   主要差异收敛在 `query profile / data tool / requestKind`，能较好验证“模板复用 + 配置覆盖”，而不需要重写主流程。

## 3. 不选择其他场景的原因

当前仓库里另一条现成场景 `payment-info-split` 不适合作为阶段 6 的第二业务样本，原因如下：

1. 它走 `direct-model`，不是当前 `agent-runtime + grounded advisory` 体系。
2. 输入是 `rawText`，不是 `opportunityId`。
3. 输出是字段拆分，不是建议类结构化结果。
4. 复用点太少，无法证明当前模板具备真正的“同类业务复用能力”。

因此，`payment-info-split` 更适合未来验证“另一类模板”，不适合当前阶段。

## 4. 与当前业务的复用关系

### 4.1 可以直接复用的部分

下列能力预计可以直接复用，不需要重写主流程：

- `grounded-structured-advisory@v1` 模板
- `validate_input`
- `authorize_scope`
- `load_reference_bundle`
- `normalize_facts`
- `select_basis_fields`
- `draft_business_output`
- `validate_output`
- `repair_output`
- `finalize_result`
- `observe_run`
- 输出 envelope 适配逻辑
- fallback / trace / rollout report 机制

### 4.2 主要通过配置覆盖的部分

`sales-opportunity-advisor-directdb` 与当前 helper 版的差异，预计应主要通过配置覆盖：

1. `scene`
   - `sales-opportunity-advisor` -> `sales-opportunity-advisor-directdb`

2. `runtimeContract`
   - `requestKind` 不同
   - request/result marker 可沿用当前模式，但值跟随 directdb 场景定义

3. `dataBindings`
   - `queryProfileRef` 需要切换到 directdb 版本

4. `toolBindings.context_fetcher`
   - 需要从 helper 数据 tool 切到 directdb runner 数据 tool

5. `nodeOverrides.fetch_business_context`
   - `toolRole` 可以继续叫 `context_fetcher`
   - 底层 `toolRef` 绑定对象不同

## 5. 差异点逐项说明

### 5.1 Prompt

V1 判断：**先复用当前业务 prompt，不要求在 `P6-T2` 单独重写。**

原因：

- helper 版与 directdb 版的业务目标相同
- 生成建议时消费的是规范化后的 facts，而不是直接消费 helper / directdb 的原始调用方式
- 只要 facts 结构保持一致，prompt 可以先复用

后续如果 directdb 场景出现专有提示词要求，再新增独立 `promptRef`，但这不应成为 `P6-T2` 的前置阻塞项。

### 5.2 Schema

V1 判断：**复用当前输出 schema。**

原因：

- 两个场景都输出相同的建议类 payload 结构
- 当前 directdb scene config 已指向同一份输出 schema 资产

### 5.3 Query Profile

V1 判断：**必须新增或切换为 directdb 专用 query profile。**

这是当前第二业务最核心的差异点。

原因：

- helper 版通过 `ContextHelper` 拉取 `rawRow`
- directdb 版通过 `DirectDbRunner` 执行缓存 SQL 模板取数
- 两者虽然都产出 `rawRow`，但底层 tool driver、约束和迁移来源不同

因此 `P6-T2` 必须为 directdb 场景补一份独立 query profile，并绑定 directdb tool。

### 5.4 Rules 文件

V1 判断：**先复用当前业务 rules 文件。**

原因：

- 当前建议规则仍然围绕销售机会阶段、预算、赢率、投标日期、竞争对手等事实字段展开
- directdb 版没有体现出与 helper 版不同的建议决策规则

如果后续 directdb 链路暴露出字段质量差异，再考虑拆出独立 rules。

### 5.5 Dictionary

V1 判断：**复用当前 dictionary。**

原因：

- 两个场景都在同一销售机会领域
- 最终都需要对相同字段做枚举映射、金额格式化、basisFields grounding

## 6. 为什么它符合“模板复用验证样本”

`sales-opportunity-advisor-directdb` 是一个“同业务目标、同输出契约、同事实结构、仅替换数据获取方式”的样本。

如果 `P6-T2` 能只通过新增业务配置和极少量资产完成接入，就可以证明：

1. 当前模板不是只为 helper 版写死的
2. 节点粒度足够支撑“换数据源不换主流程”
3. 配置协议已经能承载一类相似业务的差异

如果它仍然需要大幅重写节点或编译逻辑，就说明当前模板抽象还不够稳。

## 7. `P6-T2` 的最小实现边界

下一步 `P6-T2` 应只做下面这些事情：

1. 新增 `sales-opportunity-advisor-directdb` 的 `BusinessSkill` 配置
2. 新增 directdb 版本 `ToolDefinition`
3. 新增 directdb 版本 `QueryProfile`
4. 绑定现有模板 `grounded-structured-advisory@v1`
5. 尽量复用当前 prompt / schema / rules / dictionary

不应在 `P6-T2` 做的事情：

- 重写 `normalize_facts`
- 重写 `draft_business_output`
- 新造第二套模板
- 为 directdb 场景定制新的 graph runtime

## 8. 当前样本选择的结论

阶段 6 的第二业务样本冻结为：

- `sales-opportunity-advisor-directdb`

当前结论是：

- 这是一个足够相似、又具备关键差异的样本
- 其主要差异应由配置覆盖，而不是重写主流程
- `P6-T2` 可以以它为目标继续推进
