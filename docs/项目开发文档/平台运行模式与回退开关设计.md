# 平台运行模式与回退开关设计

## 1. 文档目标

本文档用于在进入并行迁移前，先冻结一套统一的运行模式和回退开关口径，供后续任务直接复用：

- `P1-T1` 内部 Agent Gateway 路由器
- `P1-T2` scene 运行模式配置
- `P3-T3` shadow 执行
- `P5-T1` 灰度切流
- `P5-T2` 自动 fallback

这份文档当前是“设计基线”，不是“已落地实现说明”。

## 2. 设计原则

1. 不推倒重来，默认保留当前稳定路径。
2. 运行模式必须以 scene 为粒度控制。
3. `shadow` 只做旁路观察，不直接对调用方返回新结果。
4. 自动回退只允许处理平台内部失败，不允许掩盖权限、安全和业务显式错误。
5. 任何时候都必须能一键切回 `legacy`。

## 3. 运行模式定义

V1 统一定义三种运行模式：

### 3.1 `legacy`

含义：

- 继续走当前已运行链路。
- 对 `agent-runtime` scene，仍然是：
  - API
  - Gateway
  - 退役 Agent 运行时 agent
  - 现有主 skill
  - 现有 data tool / model tool

用途：

- 默认模式
- 新平台未接入前的稳定基线
- 任何阶段的最终兜底模式

### 3.2 `shadow`

含义：

- 正式响应仍然来自 `legacy`
- 同一请求旁路执行一条候选新链路
- 候选结果只用于 trace、diff、审计和诊断

用途：

- 验证 LangGraph 兼容子图
- 验证模板化 workflow
- 不影响现网返回的情况下观察新链路行为

关键约束：

1. `shadow` 不是正式返回模式。
2. `shadow` 的结果不能直接返回给调用方。
3. `shadow` 的 request/session 必须与 `legacy` 隔离。

### 3.3 `langgraph`

含义：

- 正式响应优先由新平台编译出的 LangGraph workflow 返回
- 如命中允许自动回退的内部错误，再切回 `legacy`

用途：

- 小流量灰度
- 新平台正式接管

关键约束：

1. 只有通过 `shadow` 验证后，才能进入 `langgraph` 正式模式。
2. `langgraph` 下必须保留 `legacy` 作为回退目标。

## 4. 场景级配置字段设计

V1 建议在 scene 配置中新增设计态字段：

```json
{
  "routing": {
    "mode": "legacy",
    "shadow": {
      "enabled": false,
      "baselineMode": "legacy",
      "candidateMode": "langgraph",
      "recordDiff": true,
      "returnSource": "baseline"
    },
    "fallback": {
      "enabled": true,
      "targetMode": "legacy",
      "on": [
        "graph_compile_failed",
        "graph_timeout",
        "graph_node_failed",
        "tool_transport_failed",
        "llm_transport_failed",
        "graph_invalid_output"
      ],
      "maxPerRequest": 1
    }
  }
}
```

说明：

- `routing.mode` 是主开关。
- `routing.shadow` 用于定义旁路执行行为。
- `routing.fallback` 用于定义正式模式下的自动回退规则。

## 5. 默认值定义

V1 默认值如下：

### 5.1 默认运行模式

所有 scene 默认：

```json
{
  "routing": {
    "mode": "legacy"
  }
}
```

说明：

- 即使 scene 中没有 `routing` 字段，也应按 `legacy` 解释。
- 这样可以保证旧配置在未改造前行为不变。

### 5.2 默认 shadow 配置

默认：

- `shadow.enabled = false`
- `shadow.baselineMode = legacy`
- `shadow.candidateMode = langgraph`
- `shadow.returnSource = baseline`

### 5.3 默认 fallback 配置

默认：

- `fallback.enabled = true`
- `fallback.targetMode = legacy`
- `fallback.maxPerRequest = 1`

说明：

- 这里的“默认开启”是设计口径，不代表当前代码已经实现自动回退。
- 真正实现时，仍需要按 scene 或灰度策略显式放开。

## 6. 切换顺序与回退顺序

### 6.1 推荐切换顺序

同一个 scene 的推荐迁移顺序：

```text
legacy
  -> shadow
  -> langgraph（白名单 / 小流量）
  -> langgraph（默认）
```

解释：

1. `legacy`：当前稳定链路
2. `shadow`：旁路观察新链路
3. `langgraph` 小流量：让新链路承接正式结果
4. `langgraph` 默认：完成切流

### 6.2 推荐回退顺序

推荐回退顺序：

```text
langgraph
  -> shadow
  -> legacy
```

解释：

1. 如果新平台正式模式出现问题，先停止正式返回，退回 `shadow`
2. 如果 `shadow` 也需要关闭，最终切回 `legacy`

注意：

- 单次请求的自动 fallback 目标不是 `shadow`，而是直接 `legacy`
- `shadow` 是阶段性运行模式，不是 per-request 的自动容灾模式

## 7. 自动回退的适用范围

自动回退只允许处理“新平台内部失败”，不允许吞掉调用方应直接感知的错误。

### 7.1 允许自动回退的错误

建议仅包括：

- `graph_compile_failed`
- `graph_timeout`
- `graph_node_failed`
- `tool_transport_failed`
- `llm_transport_failed`
- `graph_invalid_output`

这些错误的共同点是：

- 错误来自新平台内部
- 旧路径大概率仍可提供服务
- 自动切回 `legacy` 不会突破安全边界

### 7.2 不允许自动回退的错误

以下错误不应自动回退：

- `INVALID_REQUEST`
- `POLICY_DENIED`
- `AUTH_FAILED`
- `OPPORTUNITY_NOT_FOUND`
- 任何字段白名单 / 脱敏 / 权限拒绝错误

原因：

1. 这些错误本身就是调用方应收到的真实结果
2. 如果因为权限或安全错误自动切回旧链路，可能绕过平台边界
3. 如果业务主键查无数据，回退旧链路也不会改变事实

## 8. Shadow 模式行为约束

为了让后续 shadow diff 可被稳定分析，V1 应冻结下面几条规则：

1. `shadow` 必须保留独立的 `request_id`
2. `shadow` 必须保留独立的 `session_key`
3. 正式响应继续返回 baseline 结果
4. `shadow` 结果必须写入 trace / diff 报告
5. `shadow` 失败不能污染正式响应

建议的记录字段：

- `baseline_request_id`
- `shadow_request_id`
- `scene`
- `baseline_mode`
- `candidate_mode`
- `diff_summary`
- `fallback_triggered`

### 9.2 当前 direct-model scene 的 V1 边界

对于 `payment-info-split` 这类 `direct-model` scene，V1 先建议：

- 暂不接入 `shadow` / `langgraph`

原因：

- 当前目标是先旁路迁移 `sales-opportunity-advisor`
- `direct-model` scene 不在本轮平台化主线内

## 10. 与后续任务的关系

这份设计文档在后续任务中的直接用途如下：

### 10.1 对 `P1-T1 / P1-T2`

提供：

- `routing.mode` 字段名
- `legacy / shadow / langgraph` 运行口径
- 默认模式定义

### 10.2 对 `P3-T3`

提供：

- shadow 的返回规则
- request/session 隔离规则
- diff 记录要求

### 10.3 对 `P5-T1 / P5-T2`

提供：

- 切流顺序
- 自动 fallback 允许范围
- 不允许 fallback 的安全边界

## 11. 当前结论

在并行迁移开始前，V1 应冻结的最小结论是：

1. 所有 scene 默认 `legacy`
2. `shadow` 只观察，不返回
3. `langgraph` 才承接正式返回
4. 自动 fallback 只处理平台内部错误
5. 任何时候都必须能切回 `legacy`

这五条是后续实现路由器、shadow 执行、灰度切流和自动回退时的统一口径。
