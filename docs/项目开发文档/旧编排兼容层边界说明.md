# 旧编排兼容层边界说明

## 1. 目标

本文档对应 `P7-T2`，用于明确一条新的默认规则：

- 旧编排仍然保留
- 但它只作为兼容执行层和回退层存在
- 新增 `agent-runtime` 业务不再允许只靠 scene-config + legacy skill 复制接入

## 2. 当前边界

### 2.1 legacy 仍然承担什么

当前 legacy 路径只保留以下职责：

1. 已模板化 scene 的显式 `legacy` 模式
2. `shadow` 模式下的 baseline 执行
3. `langgraph` 自动 fallback 目标执行器
4. 尚未模板化的 `direct-model` scene 正式执行

对应代码：

- API 入口：[routes/agent.js](/Users/gato-pm/Desktop/API_副本/routes/agent.js)
- 旧执行器：[services/runtime.js](/Users/gato-pm/Desktop/API_副本/services/runtime.js)
- 路由决策：[platform/gateway/index.js](/Users/gato-pm/Desktop/API_副本/platform/gateway/index.js)

### 2.2 legacy 不再承担什么

从当前版本开始，legacy 不再作为新增 `agent-runtime` 业务的默认接入路径。

具体规则是：

- 如果一个 scene 的 `execution.mode = agent-runtime`
- 但它没有在 [platform/skills](/Users/gato-pm/Desktop/API_副本/platform/skills) 中注册对应 `BusinessSkill`

那么 gateway 会在选路阶段直接拒绝该 scene，而不是放行它继续走旧 Gateway + old skill 编排链路。

## 3. 当前代码规则

### 3.1 scene 必须先成为 template-backed scene

[platform/gateway/index.js](/Users/gato-pm/Desktop/API_副本/platform/gateway/index.js) 现在会在 agent-runtime 选路前校验：

- 当前 scene 是否已经在 `platform/skills/*.yaml` 中注册

如果没有注册，会返回：

- `code = INVALID_REQUEST`
- `stage = scene-routing`

并带上：

- `details.requirement = template_backed_agent_runtime_scene`

### 3.2 legacy route 的语义被显式标记

当前 routePlan 会额外标记：

- `platformManagedScene`
- `legacyRole`

其中：

- 已模板化 scene 的 legacy 路径会被标记为 `compatibility`
- shadow 基线会被标记为 `compatibility-shadow-baseline`
- langgraph fallback 会被标记为 `compatibility-fallback`
- direct-model scene 保持 `legacy-primary`

这样后续日志、排障和下线评估可以清楚区分：

- “兼容层 legacy”
- “仍为主执行的 legacy”

## 5. 给后续阶段的约束

阶段 7 之后，如果要新增一个 `agent-runtime` 业务，正确路径应是：

1. 先在 [platform/templates](/Users/gato-pm/Desktop/API_副本/platform/templates) 选择或新增模板
2. 在 [platform/skills](/Users/gato-pm/Desktop/API_副本/platform/skills) 注册业务 skill
3. 在 [platform/tools](/Users/gato-pm/Desktop/API_副本/platform/tools) 注册 tool / query profile
4. 再通过 scene-config 暴露 API 和 routing

而不是：

1. 先写新的 legacy skill
2. 再只通过 scene-config 把它挂到旧 Gateway 上

## 6. 回退原则

这次收口不删除 legacy 执行器，只改变它的定位：

- 对已模板化 scene：legacy = compat / fallback
- 对未模板化 direct-model scene：legacy = primary

因此如果后续 langgraph 运行异常，仍然可以按 scene 切回 legacy，不影响对外 API 兼容性。
