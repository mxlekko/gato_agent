## 4. 当前真实工作流总览

当前 helper 场景从外部请求到最终返回的真实链路是：

```text
调用方
  -> API /api/agent/run
  -> scene 配置加载与请求校验
  -> runtime request 构造
  -> 旧 Gateway /v1/chat/completions
  -> sales-agent
  -> sales_opportunity_advisor skill
  -> ContextHelper data tool
     -> 读取 skill 中的 helper 查询脚本路径和查询业务定义
     -> 复用或生成查询脚本
     -> SQL Server 查询 rawRow
  -> skill 读取字典 / 规则 / schema
  -> skill 内完成字段清洗、字典映射、建议生成
  -> ModelTool 校验结构化输出
  -> skill 输出 wrapped JSON
  -> API 解析 result markers 后统一回包
  -> 调用方
```

## 6. 真实运行证据（2026-04-12 观测样本）

本地已有一次真实会话样本：

- [旧共享运行时目录/agents/sales-agent/sessions/ac38c23a-f2fd-4903-8981-3f2b5db7abcd.jsonl](旧共享运行时目录/agents/sales-agent/sessions/ac38c23a-f2fd-4903-8981-3f2b5db7abcd.jsonl)

从这份会话样本可以确认当前实际执行顺序有以下特征：

1. Gateway 实际调用的模型提供方是 `moonshot / kimi-k2.5`。
2. agent 首先尝试读取错误路径 `skills/sales_opportunity_advisor/SKILL.md`，失败后改为正确路径 `skills/sales-opportunity-advisor/SKILL.md`。
3. skill 在第一轮工具动作中，把 helper 调用与字典 / 规则 / schema 读取并行发起。
4. helper 实际只返回 `data.rawRow`，没有返回 `profile` 或 `facts`。
5. 随后由 skill 组织候选 payload，再调用 ModelTool 校验。
6. 最终返回的是带 result markers 的 wrapped JSON，而不是裸 payload。

这说明当前“业务编排”和“事实整理”确实在 skill 内部完成，而不是在 data tool 或 model tool 内完成。

## 7. 当前组件职责边界

### 7.1 API 层

负责：

- 接收外部请求
- 校验 `scene` 和 `bizParams`
- 读取 scene 配置
- 生成 `requestId`
- 构造 runtime request
- 调用 Gateway
- 解析 wrapped result 并统一回包

不负责：

- 字段清洗
- 业务建议生成
- schema 细粒度校验

### 7.2 sales-agent / skill 层

负责：

- 识别 machine request
- 解析 request markers 里的 JSON
- 调用 data tool
- 读取字典、规则、schema
- 在 skill 内完成字段映射、事实整理、建议生成
- 调用 model tool 校验
- 输出 wrapped JSON

不负责：

- 直接管理数据库连接
- 自己执行原生 SQL

### 7.3 ContextHelper

负责：

- 接收 `requestId`、`opportunityId`
- 读取 skill 中的 helper 查询脚本路径和查询业务定义
- 复用或生成本地查询脚本
- 执行参数化 SQL
- 返回 `rawRow`

不负责：

- 生成业务建议
- 返回 `profile` 或 `facts`
- 结构化输出校验

### 7.4 ModelTool

负责：

- 校验 `payload` 是否满足 `schema`
- 进行轻量规范化，例如去重、截断、trim

不负责：

- 重新生成业务建议
- 参与查询服务
- 校验业务事实是否真的来自 `rawRow`

## 9. 当前基线下的关键风险

### 9.1 路径耦合风险

虽然当前工作目录是 `/Users/gato-pm/Desktop/API_副本`，但 helper scene 和 skill 中大量路径仍然指向：

- `/Users/gato-pm/Desktop/API/...`

这意味着：

- 当前系统真实运行依赖“原始目录”中的资产位置
- 后续平台化迁移如果直接改目录，容易把运行链路切断

### 9.2 事实 grounding 风险

当前 skill 会基于 `rawRow`、字典和规则草拟建议，但 ModelTool 只校验 schema，不校验“建议是否真的被 `rawRow` 支撑”。

结合 2026-04-12 的会话样本，可见当前链路存在以下事实风险：

- `rawRow` 中未必包含 `budgetConfirmed`，但最终建议可能仍提到“预算尚未确认”
- 字典枚举与原始值不一致时，skill 可能仍输出看似合理但不完全可追溯的建议

这部分是后续迁移到 LangGraph 时必须单独拆分和校验的高风险节点。

### 9.3 Scene 配置与实际执行的双重事实来源

当前存在两个“事实来源”：

1. 本仓库的 scene config
2. `旧共享运行时目录` 中 agent 的 `AGENTS.md + SKILL.md`

scene config 提供的是元数据和接口说明；真正的运行编排仍然由 skill 文本决定。

这也是后续需要把“模板、业务 skill、tool registry、compiler”分离出来的直接原因。
