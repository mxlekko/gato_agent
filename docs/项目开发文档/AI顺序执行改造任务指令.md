# AI 顺序执行改造任务指令

你现在负责执行“MySQL 配置中心化改造”任务。

执行前必须先读取以下文件：

- `MySQL配置中心化改造系统架构与任务清单.md`
- `MySQL配置中心化改造执行看板.md`

## 1. 总目标

按既定顺序逐项完成改造任务，不跳任务，不并行推进多个任务，不把运行时直接切成查 MySQL。

生产方案必须始终遵守：

**MySQL 做 Source of Truth，发布生成 release bundle，运行时读取已发布 bundle。**

## 2. 执行算法

你每一轮都必须严格执行下面的算法：

1. 打开 `MySQL配置中心化改造执行看板.md`
2. 从“任务总表”第一行开始向下扫描
3. 找到**第一个状态不是 `DONE` 的任务**
4. 根据该任务状态执行：
   - 如果是 `DOING`：继续完成该任务
   - 如果是 `TODO`：先把状态改成 `DOING`，再开始执行
   - 如果是 `BLOCKED`：停止执行，输出阻塞原因、影响范围、需要用户提供的信息
5. 每次只允许执行这一个任务
6. 不允许跳到后面的任务
7. 完成后必须把该任务状态改成 `DONE`
8. 同时回填该任务卡片中的：
   - 开始时间
   - 完成时间
   - 改动文件
   - 新增文件
   - 验证命令
   - 风险 / 备注
   - 回退方式
9. 更新完成后，本轮结束
10. 下一轮再次从任务总表顶部扫描，不允许凭记忆续做后续任务

## 3. 必须遵守的改造约束

1. 先抽象后迁移
   - 先引入 `config-store`
   - 再逐步替换调用点

2. 先控制面后运行面
   - 先改控制台读写 MySQL 草稿
   - 再做 release manager / bundle renderer
   - 最后切 runtime 到 active bundle

3. 禁止一步切到“运行时直查 MySQL”

4. 禁止跨模块大爆炸式重构

5. 所有写 MySQL 的动作都必须生成 revision

6. 所有发布动作都必须生成 release_id

7. 运行时请求路径不允许读取草稿

8. 如果当前任务未完成，不得启动下一任务

- 风险说明：
- 回退方式：

### 下一步

- 下次启动时，请重新扫描执行看板，继续处理第一个未完成任务

## 5. 失败处理规则

如果当前任务无法完成：

1. 把该任务状态更新为 `BLOCKED`
2. 在任务卡片中写清：
   - 阻塞原因
   - 当前已完成到哪一步
   - 需要什么输入或决策
   - 推荐下一步
3. 停止继续处理后续任务
