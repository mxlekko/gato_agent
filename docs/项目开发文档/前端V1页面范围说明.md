# 前端 V1 页面范围说明

## 1. 文档目标

本文档对应 `FE0-T1`，用于冻结前端控制台 V1 的最小页面范围。

这份文档只回答三个问题：

1. V1 要做哪些页面
2. 每类页面解决什么问题、面向谁使用
3. V1 明确不做哪些能力

本文档不展开接口字段和数据结构细节，接口映射在后续 `FE0-T2` 单独冻结。

## 2. 产品定位

前端 V1 的定位固定为：

- 面向内部的“Agent 业务平台控制台”
- 主要服务迁移、联调、排障、灰度和受控配置查看
- 不是外部调用方页面
- 不是低代码拖拽编排器
- 不是绕过后端校验的在线配置入口

V1 原则：

1. 先只读、后受控写
2. 先支持联调和排障，后支持运营化
3. 不阻塞后端主线迁移
4. 前端工程故障时，不影响现有 API 对外提供能力

## 3. V1 目标用户

V1 默认面向四类内部用户：

1. 后端研发
   - 查看 scene / workflow 映射
   - 发起联调请求
   - 排查节点失败
2. 测试 / 联调同学
   - 回放请求
   - 查看 Run 结果和 shadow 差异
3. 平台治理同学
   - 查看配置、校验结果、编译预览
   - 观察灰度切流状态
4. 值班 / 运维同学
   - 通过 trace / rollout / fallback 信息定位问题

## 4. V1 信息架构

V1 控制台的信息架构固定为五个一级域：

1. `Scenes`
   - 看有哪些业务 scene
   - 看 scene 如何映射到 workflow 模板
2. `Debug`
   - 发起一次请求，做最小 API 联调
3. `Runs`
   - 看运行记录、结果、shadow 差异、trace 详情
4. `Configs`
   - 看模板、skill、tool、query、编译预览和校验结果
5. `Rollout`
   - 看灰度、fallback、下线门槛相关信息

建议页面路由骨架如下：

```text
/scenes
/scenes/:scene
/debug/run-once
/runs
/runs/:runId
/runs/:runId/shadow
/traces/:traceId
/configs
/configs/templates/:name
/configs/skills/:name
/configs/validate
/configs/compile-preview
/rollout
```

说明：

- 上述是 V1 范围骨架，不代表 FE1 就一次性做完
- 具体分页按阶段拆分，见下文

## 5. V1 页面清单

### 5.1 FE1 范围：最小只读与联调

#### 1. Scene / Workflow 浏览页

目标：

- 让用户快速理解“当前有哪些 scene、它们映射到哪个模板、当前运行模式是什么”

页面内容：

- Scene 列表
- Scene 基本说明
- 当前 routing mode / allowed modes
- 对应 template / skill / tool / query 绑定概览

数据来源类别：

- scene 配置
- 模板配置
- business skill 配置
- tool / query 配置

定位：

- 只读页

#### 2. 单次请求调试页

目标：

- 在页面中发起一次 `POST /api/agent/run`，用于最小联调

页面内容：

- 选择 scene
- 输入 bizParams
- 发起请求
- 查看 response envelope
- 查看请求路由模式和 requestId

数据来源类别：

- 真实 API
- 联调 mock 数据

定位：

- 调试页

### 5.2 FE2 范围：运行记录、shadow、trace

#### 3. Run 列表页

目标：

- 查看最近运行记录和运行状态，支持按 scene / mode / success 过滤

页面内容：

- Run 列表
- scene / requested mode / effective mode
- success / failure
- requestId / traceId
- duration / fallback / shadow 状态

数据来源类别：

- 运行记录
- rollout / trace 聚合结果

定位：

- 只读页

#### 4. Run 详情页

目标：

- 查看一次运行的最终请求、响应、错误和关键上下文

页面内容：

- 请求摘要
- response envelope
- error code / stage
- fallback 信息
- 关联 trace / shadow 链接

数据来源类别：

- 单次运行详情
- trace 上下文

定位：

- 只读页

#### 5. Shadow 对比页

目标：

- 让用户直接看到 baseline 与 shadow 的结果差异，而不是只看最终是否成功

页面内容：

- baseline / shadow 请求摘要
- HTTP / envelope / stable fields 对比
- 差异列表
- diff pass / difference count

数据来源类别：

- shadow diff 报告
- run 对比数据

定位：

- 排障页

#### 6. Trace 详情页

目标：

- 节点级定位问题，看到每个节点的输入摘要、输出摘要、耗时和异常

页面内容：

- node run 时间线
- 每个节点的 status / duration
- error code / stage
- tool 调用和 fallback 关联信息

数据来源类别：

- workflow state 中的 `artifacts.node_runs`
- trace / audit 记录

定位：

- 排障页

### 5.3 FE3 范围：配置查看、预校验、灰度

#### 7. 配置查看与差异页

目标：

- 浏览模板、skill、tool、query 配置，并查看版本差异

页面内容：

- 模板详情
- skill 详情
- tool / query 详情
- diff 视图

数据来源类别：

- 配置文件
- 配置 registry 聚合结果

定位：

- 只读页

#### 8. 配置校验与编译预览页

目标：

- 在不真正生效的情况下，查看配置是否合法、模板和业务配置合并后是什么图

页面内容：

- 配置校验结果
- issue 列表
- 编译后的 workflow 概览
- node order / edge 概览

数据来源类别：

- 配置校验器
- workflow compiler 输出

定位：

- 受控操作页

#### 9. 灰度切流操作页

目标：

- 在受控前提下查看和调整 scene 的 routing / cutover 状态

页面内容：

- 当前 routing mode
- allowed modes
- cutover 命中策略摘要
- fallback 观察摘要

数据来源类别：

- scene routing 配置
- rollout 聚合结果

定位：

- 受控操作页

说明：

- V1 不要求前端直接修改线上文件
- 即使存在操作入口，也必须通过后端校验、审计和受控发布链路

## 6. V1 明确不做的范围

以下内容明确不在 V1 范围内：

1. 可视化拖拽编排器
2. 在线自由插入 / 删除 / 重排节点
3. 在线自由编辑 prompt / SQL / 脚本原文
4. 直接向生产环境发布配置
5. 生产环境直接发布配置
6. 越过后端策略层直接调用内部 tool
7. 面向外部调用方的业务页面
8. 多租户运营后台、权限中心、审批流中心

## 7. 页面优先级

V1 页面优先级固定为：

1. `P0`
   - Scene / Workflow 浏览页
   - 单次请求调试页
2. `P1`
   - Run 列表页
   - Run 详情页
   - Shadow 对比页
   - Trace 详情页
3. `P2`
   - 配置查看与差异页
   - 配置校验与编译预览页
   - 灰度切流操作页

这意味着：

- FE1 先做 `P0`
- FE2 补齐 `P1`
- FE3 再做 `P2`

## 8. 数据来源边界

虽然接口映射在 `FE0-T2` 冻结，但 V1 先固定下面三类数据来源边界：

1. 必须真实对接
   - `POST /api/agent/run` 最小联调
   - 真实 rollout / trace / run 数据
2. 可先 mock
   - Scene / workflow 概览列表
   - 配置差异预览草图
   - 编译预览展示壳
3. 必须受控
   - 灰度切流操作
   - 配置校验与编译预览

## 9. 页面范围完成判定

`FE0-T1` 完成时，应满足：

1. 页面清单已经按阶段冻结
2. 只读页、调试页、trace 页、配置查看页边界清晰
3. 明确写出 V1 不做拖拽编排和生产直发
4. 每类页面都有对应的数据来源类别说明

## 10. 给 FE0-T2 的直接输入

下一步 `FE0-T2` 应基于本文档继续展开：

- 每个页面具体依赖哪些接口
- 哪些接口可先 mock
- 哪些接口必须真实提供
- 每个页面最小返回结构是什么
