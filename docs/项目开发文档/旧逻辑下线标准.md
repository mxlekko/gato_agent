# 旧逻辑下线标准

## 1. 目标

本文档对应 `P7-T3`，用于明确：

- 旧逻辑什么时候只能作为兼容层存在
- 什么条件下可以把某个 scene 的默认执行从 `legacy` 切到 `langgraph`
- 什么条件下可以关闭某个 scene 的显式 `legacy` 模式
- 什么条件下才允许删除 `agent-runtime` 旧执行器代码

目标不是立即删除旧代码，而是给“何时退场”设定明确门槛，避免长期双轨维护失控。

## 2. 适用范围

本标准只适用于：

- `execution.mode = agent-runtime` 的 scene
- 且该 scene 已经在 [platform/skills](/Users/gato-pm/Desktop/API_副本/platform/skills) 中注册 `BusinessSkill`

当前适用场景：

- `sales-opportunity-advisor`
- `sales-opportunity-advisor-directdb`

当前不适用：

- `payment-info-split`

原因：

- 它仍是 `direct-model` 旧链路场景，还没有模板化 workflow

## 3. 下线对象分层

旧逻辑不是一次性整体删除，而是分三层退场：

### 3.1 第 1 层：切换 scene 默认执行

含义：

- scene-config 的主运行模式从 `legacy` 切到 `langgraph`
- 但 `legacy` 仍作为 fallback 和人工回退开关存在

这是最早可以发生的“退场动作”。

### 3.2 第 2 层：关闭 scene 级显式 `legacy`

含义：

- scene 不再允许配置为显式 `legacy`
- 仅保留受控 fallback

这一步意味着该 scene 已经不再把旧链路当作日常运行方式。

### 3.3 第 3 层：删除全局 `agent-runtime` 旧执行器

含义：

- [services/runtime.js](/Users/gato-pm/Desktop/API_副本/services/runtime.js) 中面向 `agent-runtime` 的 legacy 主执行职责可以退场
- [services/runtime-message.js](/Users/gato-pm/Desktop/API_副本/services/runtime-message.js) 和 [services/response-parser.js](/Users/gato-pm/Desktop/API_副本/services/response-parser.js) 中仅服务于 legacy agent-runtime 的路径可以进入删除计划

这一步必须最后发生。

## 4. 统一判定指标

下线标准统一使用当前已存在的 rollout / regression 指标，不额外发明新口径。

### 4.1 指标来源

- rollout 聚合：
  - [scripts/build_rollout_report.js](/Users/gato-pm/Desktop/API_副本/scripts/build_rollout_report.js)
  - [platform/trace/rollout-report.js](/Users/gato-pm/Desktop/API_副本/platform/trace/rollout-report.js)
- baseline 回归：
  - [scripts/run_baseline_regression.js](/Users/gato-pm/Desktop/API_副本/scripts/run_baseline_regression.js)
  - [tests/regression/README.md](/Users/gato-pm/Desktop/API_副本/tests/regression/README.md)

### 4.2 核心指标

| 指标 | 来源字段 | 用途 |
|---|---|---|
| 成功率 | `rates.successRate` | 判断新路径是否稳定可用 |
| fallback 比例 | `rates.fallbackRatio` | 判断 langgraph 是否仍频繁回退到旧逻辑 |
| schema 失败率 | `rates.schemaFailureRate` | 判断结构化输出是否稳定 |
| p95 延迟 | `latency.p95DurationMs` | 判断新路径是否明显慢于可接受范围 |
| shadow diff 通过率 | `shadowDiff.diffPassRate` | 判断新旧结果是否一致 |
| 失败分布 | `failures.byCode` / `failures.byStage` | 判断是否仍存在集中性故障 |

### 4.3 业务错误处理口径

以下业务错误不直接阻断下线判定，但必须保持语义正确：

- `OPPORTUNITY_NOT_FOUND`
- 权限拒绝类错误
- 请求参数错误类错误

判断重点是：

- 是否按预期返回
- 是否没有被错误地 fallback 掩盖

## 5. 下线门槛

### 5.1 场景默认切到 `langgraph` 的门槛

某个 template-backed scene 要把默认执行从 `legacy` 切到 `langgraph`，必须同时满足：

1. 稳定时间
   - 连续 `7` 个自然日保持可观测运行
2. 业务覆盖
   - 该 scene 至少有 `100` 次 `langgraph` 实际运行样本
3. 成功率
   - `successRate >= 0.98`
4. fallback 比例
   - `fallbackRatio <= 0.02`
5. schema 失败率
   - `schemaFailureRate <= 0.01`
6. shadow 一致性
   - 若存在 shadow 对比样本，则 `shadowDiff.diffPassRate >= 0.98`
7. 回归
   - baseline regression 全部通过
8. 事故
   - 最近 `7` 天无由 langgraph 路径导致的 P1 / P2 事故

### 5.2 关闭某个 scene 显式 `legacy` 的门槛

某个 scene 要从“可显式 legacy”进入“仅 fallback 保留”的状态，必须同时满足：

1. 已经以 `langgraph` 为默认执行至少 `14` 个自然日
2. 最近 `14` 天至少 `300` 次 langgraph 运行样本
3. `successRate >= 0.99`
4. `fallbackRatio <= 0.005`
5. `schemaFailureRate <= 0.005`
6. 若存在 shadow 对比，则最近 `14` 天 `shadowDiff.diffPassRate >= 0.99`
7. 没有未关闭的高优先级兼容差异单
8. 已完成一次“手动切回 legacy 再切回 langgraph”的演练，并确认回退开关可用

### 5.3 删除全局 `agent-runtime` 旧执行器的门槛

要删除 `agent-runtime` 旧执行器代码，必须同时满足：

1. 至少 `2` 个 `agent-runtime` 业务已经模板化并稳定运行于新平台
2. 这些业务均已关闭显式 `legacy` 至少 `14` 个自然日
3. 最近 `30` 天跨 scene 聚合的：
   - `successRate >= 0.99`
   - `fallbackRatio <= 0.005`
   - `schemaFailureRate <= 0.005`
4. 最近 `30` 天没有依赖 legacy agent-runtime 主路径处理的新需求上线
5. 所有仍未迁移 scene 都已明确归类为：
   - `direct-model` 旧链路保留
   - 或已冻结、不会继续扩展
6. 已确认：
   - 没有生产流量依赖显式 `legacy` agent-runtime 模式
   - fallback 命中量已低到只剩异常应急用途

## 6. 下线前检查项

每次准备执行下线动作前，至少检查以下项目：

1. 配置层
   - scene 已在 [platform/skills](/Users/gato-pm/Desktop/API_副本/platform/skills) 中注册
   - scene 的 tool / query / asset 配置已通过校验
2. 运行层
   - rollout report 已生成
   - alert 结果已经人工确认
3. 回归层
   - baseline regression 全通过
   - 至少验证 success / 参数错误 / 业务未找到 三类场景
4. 兼容层
   - fallback 仍可用
   - `legacyRole` 日志仍可观测
5. 运维层
   - 已记录回退负责人
   - 已定义回退窗口和回退步骤

## 7. 推荐执行命令

### 7.1 生成 rollout 报告

```bash
node scripts/build_rollout_report.js \
  --input platform/tests/fixtures/rollout/sample-events.jsonl \
  --batch-id sample-rollout \
  --output /tmp/sample-rollout-report.json
```

### 7.2 按下线门槛执行告警校验

下面这组参数可作为“默认切到 langgraph”门槛的机器化近似检查：

```bash
node scripts/build_rollout_report.js \
  --input <your-trace-events.jsonl> \
  --batch-id <scene-cutover-check> \
  --min-success-rate 0.98 \
  --max-p95-ms 3000 \
  --max-schema-failure-rate 0.01 \
  --max-fallback-ratio 0.02 \
  --min-shadow-diff-pass-rate 0.98 \
  --fail-on-alert
```

### 7.3 运行 baseline 回归

```bash
node scripts/run_baseline_regression.js
```

## 8. 明确不允许的情况

以下情况禁止推动旧逻辑下线：

1. 只有一个模板化业务稳定，尚未达到“至少两个业务”覆盖要求
2. fallback 仍经常命中，只是因为 legacy 能兜住所以表面成功
3. shadow diff 仍持续存在未解释差异
4. 新路径的业务错误语义尚未和旧路径对齐
5. 仍需要通过新增 legacy skill 的方式接入业务

## 9. 回退原则

即使满足门槛，下线动作也必须保持可回退：

- scene 默认切换后的前一段观察期内，保留 fallback
- 关闭显式 `legacy` 前，先保留快速配置回退方式
- 删除 agent-runtime 旧执行器前，先冻结一个可恢复分支或 tag

结论上，旧逻辑的退场顺序应是：

1. 不再承接新业务
2. 不再作为默认执行
3. 不再允许显式启用
4. 最后才删除代码
