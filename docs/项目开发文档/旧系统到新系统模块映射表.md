# 旧系统到新系统模块映射表

## 1. 文档目标

本文档对应阶段 7 的 `P7-T1`，用于回答一个很具体的问题：

- 当前哪些旧模块已经被新平台覆盖
- 哪些旧模块只是被新平台包装复用
- 哪些模块仍然是共享底座，暂时不应视为“已迁移完成”
- 哪些模块还完全停留在旧链路

本文档最初对应阶段 7 的并行迁移基线。2026-05-06 退役 Agent 运行时 退场复验后，agent-runtime 主链路已经收口到项目内 LangGraph/runtime；本文保留旧新模块映射口径，但“当前状态”以本节和后续表格中的 2026-05-06 更新为准。

## 2. 迁移状态定义

为避免“看起来像迁移了，其实只是换了入口”的误判，本文档统一使用四种状态：

- `已覆盖`
  - 新平台已经有明确对应模块，并且该职责应以新平台实现为主
- `包装复用`
  - 新平台已把旧能力纳入统一编排，但底层仍依赖旧服务或旧实现
- `共享底座`
  - 旧新两套路径共同依赖，短期不是替换对象
- `旧链路保留`
  - 当前还没有对应的新平台实现，仍只走旧执行方式

## 3. 总体结论

截至当前版本，可以把系统分成三块：

1. 对外 API 和 scene 契约层：
   - 仍由 [routes/agent.js](/Users/gato-pm/Desktop/API_副本/routes/agent.js)、[services/scene-config.js](/Users/gato-pm/Desktop/API_副本/services/scene-config.js)、[services/request-validation.js](/Users/gato-pm/Desktop/API_副本/services/request-validation.js) 维持兼容
   - 这是 `共享底座`，不是直接删除目标

2. 面向 `sales-opportunity-advisor` / `sales-opportunity-advisor-directdb` / `sales-opportunity-smart-entry` 的业务编排层：
   - 已由 [platform/templates/grounded-structured-advisory.v1.yaml](/Users/gato-pm/Desktop/API_副本/platform/templates/grounded-structured-advisory.v1.yaml)、[platform/skills/sales-opportunity-advisor.v1.yaml](/Users/gato-pm/Desktop/API_副本/platform/skills/sales-opportunity-advisor.v1.yaml)、[platform/skills/sales-opportunity-advisor-directdb.v1.yaml](/Users/gato-pm/Desktop/API_副本/platform/skills/sales-opportunity-advisor-directdb.v1.yaml)、[platform/compiler/compile-workflow.js](/Users/gato-pm/Desktop/API_副本/platform/compiler/compile-workflow.js)、[platform/runtime/graphs/index.js](/Users/gato-pm/Desktop/API_副本/platform/runtime/graphs/index.js) 和各节点承接
   - 这是当前最主要的 `已覆盖 / 包装复用` 区域；三条销售场景均已 `routing.mode=langgraph` 且 `langgraphCutover.requestPercentage=100`

3. 非模板化执行边界：
   - agent-runtime legacy 退役 Agent 运行时 执行已经退役；[services/runtime.js](/Users/gato-pm/Desktop/API_副本/services/runtime.js) 对 agent-runtime legacy 只保留显式拒绝和 direct-model 旧边界
   - [services/direct-model.js](/Users/gato-pm/Desktop/API_副本/services/direct-model.js) 仍承接 `payment-info-split`、`special-custom-product-solution` 这类 direct-model 场景；这里的 `legacy` 表示既有 direct-model 执行边界，不表示 旧 Gateway
   - [services/runtime-message.js](/Users/gato-pm/Desktop/API_副本/services/runtime-message.js) 与 [services/response-parser.js](/Users/gato-pm/Desktop/API_副本/services/response-parser.js) 仍保留 API envelope、markers、payload 校验等兼容能力，但不再作为 旧 Gateway 主调用链

## 7. 给 `P7-T2` 的直接结论

2026-05-06 现状下，阶段 7 的“旧编排降级”已经完成到可上线守门口径：

- agent-runtime legacy 退役 Agent 运行时 执行已退役
- 新业务默认入口固定到 `platform/templates + platform/skills + platform/tools + platform/compiler + platform/runtime`
- `payment-info-split` / `special-custom-product-solution` 属于 direct-model 旧边界，不经过 旧 Gateway
- 上线前继续以 `node scripts/scan_retired_runtime_dependencies.js --fail-on-runtime-blocker --fail-on-config-blocker` 和 `npm run regression:no-retired-runtime` 守门

也就是说，后续清理重点不再是迁移 agent-runtime 主链路，而是维护无 退役 Agent 运行时 blocker 的守门命令，并逐步压低历史文档和旧回归输出中的审计命中。
